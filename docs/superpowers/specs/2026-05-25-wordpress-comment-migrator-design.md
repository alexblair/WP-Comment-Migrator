# WordPress 评论迁移插件 - 产品设计文档

> 创建日期：2026-05-25
> 目标环境：PHP 7.0 / MySQL 5.7

---

## 1. 用户需求场景

| 场景 | 说明 | 优先级 |
|------|------|--------|
| A：一对多迁移 | 将一篇源文章的全部评论迁移到另一篇目标文章 | P0 |
| B：多对一合并 | 将多篇源文章的评论合并到同一篇目标文章 | P0 |
| C：选择性迁移 | 只勾选部分评论进行迁移（按状态/日期/手动筛选） | P0 |
| D：批量映射 | 批量配置多组 Source→Target 映射关系，一次性执行 | P0 |

每个场景均支持**回滚撤销**。

## 2. 技术约束

### PHP 7.0 限制
- ❌ 禁止：可空类型 `?int`（改用默认值 `= null`）
- ❌ 禁止：`void` 返回类型
- ❌ 禁止：类常量可见性 `public const`（PHP 7.1+）
- ❌ 禁止：多异常捕获 `catch (A | B)`
- ❌ 禁止：方括号 list 语法 `[$a, $b] = ...`（使用 `list()`）
- ❌ 禁止：`?->` nullsafe 运算符
- ✅ 允许：标量类型声明 `int $id`、`string $name`
- ✅ 允许：返回类型声明 `: array`
- ✅ 允许：null 合并运算符 `??`
- ✅ 允许：匿名类

### MySQL 5.7 限制
- ❌ 禁止：CTE `WITH ... AS`
- ❌ 禁止：窗口函数 `ROW_NUMBER()` 等
- ❌ 禁止：CHECK 约束强制执行
- ✅ 允许：JSON 函数
- ✅ 允许：`utf8mb4`
- ✅ 允许：`CURRENT_TIMESTAMP`、`ON UPDATE CURRENT_TIMESTAMP`

## 3. 插件架构

### 3.1 文件结构

```
wp-content/plugins/comment-migrator/
├── comment-migrator.php              # 主文件：插件头、常量定义、加载入口
├── uninstall.php                     # 卸载钩子：询问是否保留数据表
├── includes/
│   ├── class-admin.php               # 后台菜单注册、页面路由、Tab 切换
│   ├── class-list-table.php          # WP_List_Table 评论列表
│   ├── class-history-table.php       # WP_List_Table 迁移历史列表
│   ├── class-migration.php           # 迁移核心逻辑：执行迁移、更新计数
│   ├── class-rollback.php            # 回滚逻辑：从日志恢复数据
│   ├── class-batch.php               # 批量映射解析与执行
│   └── class-install.php             # 插件激活/停用：建表、删表
└── readme.txt                        # WordPress 插件标准 readme
```

### 3.2 自定义数据库表

```sql
CREATE TABLE wp_cmt_migration_logs (
    id BIGINT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
    migrate_key VARCHAR(32) NOT NULL COMMENT '迁移唯一标识，用于回滚',
    source_post_id BIGINT UNSIGNED NOT NULL COMMENT '源文章ID',
    target_post_id BIGINT UNSIGNED NOT NULL COMMENT '目标文章ID',
    comment_count INT UNSIGNED NOT NULL DEFAULT 0 COMMENT '迁移评论数',
    comment_data LONGTEXT NOT NULL COMMENT '迁移前完整数据JSON（用于回滚）',
    is_batch TINYINT(1) DEFAULT 0 COMMENT '是否批量映射操作',
    created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
    INDEX idx_migrate_key (migrate_key),
    INDEX idx_created_at (created_at)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci;
```

### 3.3 卸载行为

- 插件在 WordPress 后台点击**删除**时，通过 `uninstall.php` 拦截
- 弹出确认框："是否同时删除插件创建的 `wp_cmt_migration_logs` 数据表？保留数据表可保留历史迁移记录。"
- 用户可选择"保留数据表"或"一并删除"

## 5. 核心逻辑设计

### 5.1 迁移流程

```
用户勾选评论 → 选择目标文章 → 确认执行
  │
  ├─ 1. 备份：读取每条评论原始数据，序列化为 JSON
  ├─ 2. 记录：写入 migration_logs 表（含备份数据）
  ├─ 3. 执行：UPDATE wp_comments SET comment_post_id = target
  ├─ 4. 计数：更新源/目标文章的 comment_count
  └─ 5. 反馈：返回成功/失败结果
```

### 5.2 回滚流程

```
用户选择历史记录 → 点击回滚 → 确认
  │
  ├─ 1. 读取：从 migration_logs 读取 comment_data JSON
  ├─ 2. 还原：恢复每条评论的 comment_post_id 及所有原始字段
  ├─ 3. 计数：重新计算源/目标文章的 comment_count
  ├─ 4. 删除日志：标记该条日志为已回滚（或删除）
  └─ 5. 反馈：返回回滚结果
```

### 5.3 评论计数更新

- WordPress 评论计数存储在 `wp_posts.comment_count`
- 每次迁移和回滚后通过 `wp_update_comment_count($post_id)` 刷新

### 5.4 错误处理

- 事务性：单条评论迁移失败不影响其他评论
- 逐条执行：每条评论独立 UPDATE，可精确报告成功/失败
- 失败记录：记入 PHP error log，页面显示友好提示

### 5.5 边界情况处理

- **评论已被删除：** 若回滚时某条评论已被用户手动删除，跳过该条，继续处理其余评论，最终报告跳过的数量
- **超大批量选择：** "全选所有匹配结果"超过 1000 条时，采用分批异步处理（每批 100 条），避免 PHP/MySQL 超时
- **源=目标相同：** 如果用户选择的目标文章与源文章相同，拦截并提示错误
